# Canary Injection

## 1. The Problem

Most injection detectors infer attacks probabilistically, which can create false positives and miss subtle manipulations. Teams need a deterministic signal that confirms an agent followed malicious hidden instructions.

## 2. How AgentShield Solves It

AgentShield uses session-scoped canary tokens and scans model/tool outputs for token echo. A canary hit is treated as high-confidence manipulation and immediately escalates to block semantics.

## 3. Technical Detail

- Token format: `SHIELD-VERIFY-{16 uppercase hex chars}`
- Token generation: cryptographically random (`secrets.token_hex(8)`)
- Stored in logs/events as hash only (`token_hash`), never as raw token value
- Trigger surfaces in Phase 4B:
  - `LLM_RESPONSE`
  - `TOOL_CALL_COMPLETE`
- Rotation: every `canary_rotation_sessions` injections (default `50`)

Instruction template is generated by `build_canary_instruction()` and can be injected by adapters before model calls.

## 4. Code Example

```python
from uuid import uuid4

from agentshield.canary.system import CanarySystem
from agentshield.config import AgentShieldConfig

config = AgentShieldConfig(canary_enabled=True, canary_rotation_sessions=50)
canary = CanarySystem(config)
session_id = uuid4()

canary.initialize_session(session_id)
instruction = canary.get_canary_instruction(session_id)
print(instruction)
```

## 5. Limitations and Future Work

- Current scanning is post-response detection unless adapter injection is enabled in your runtime path.
- Canary design must remain secret and rotated to avoid model memorization effects.
- Future work includes richer placement strategies and cross-turn canary lineage analytics.
